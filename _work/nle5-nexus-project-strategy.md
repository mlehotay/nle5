# NLE5 Infrastructure and Situation-Learning Strategy

This document explains the intended architecture and long-term direction of the NLE5 project.

It complements:

* `AGENTS.md`
* `nle5-codex-copypasta.md`
* `nexus-alignment.md`
* the upstream NLE paper and associated implementation

This document clarifies:

* project boundaries
* infrastructure layering
* the relationship between NLE5 and Nexus
* the role of Gym-style interfaces
* the intended long-term learning direction
* the IER-informed strategy centered around situations, affordances, and variables

This is not a formal specification and should not be interpreted as a rigid roadmap.

The current priority remains modest:

```text id="1xhhpv"
make the bridge work
```

---

# 1. Layer Separation

The ecosystem is intentionally decomposed into independent layers.

```text id="gpr7eu"
Nexus
  ↓
NetHack-compatible world/runtime
  ↓
NLE5 bridge
  ↓
Python environment / Gym interface
  ↓
Rollouts, datasets, tooling
  ↓
Agents and learning systems
```

The dependency direction matters.

The engine should not depend on the learning stack.

Nexus should remain meaningful even if no machine-learning agents ever exist.

NLE5 should remain useful even if the eventual agents are simple.

---

# 2. Nexus

Nexus is the long-term world/runtime project.

It is not fundamentally an RL project.

Its responsibilities include:

* world simulation
* maps
* entities
* procedural generation
* interaction systems
* tiny-world experimentation
* alternate topologies and mechanics
* experimental environments

The current alignment document correctly keeps Nexus and NLE5 loosely coupled.

The intended relationship is:

```text id="jtfj0w"
Nexus first:
  boot a tiny world for a human player

NLE5 first:
  expose a controllable process to Python

Integration later:
  run NLE5 against Nexus-derived worlds
```

This separation is important because learning infrastructure should not prematurely distort engine design.

---

# 3. NLE5

NLE5 exists to provide a modern NetHack 5-compatible control and observation bridge.

The early milestones are intentionally small in scope.

The initial adapter only needs:

* reset
* step
* observations
* action injection
* short rollouts
* debugging visibility

As the alignment document states:

> “The first successful test should be `reset`, one `step`, and observation sanity, not training.”

This is deliberate.

The early infrastructure should prioritize:

* inspectability
* replayability
* modularity
* determinism where possible
* instrumentation surfaces
* observation clarity

before pursuing advanced agent behavior.

---

# 4. Gym Interfaces

The Gym layer is important but should not be mistaken for the intelligence itself.

A Gym environment provides:

```text id="a7jlwm"
observation
→ action
→ transition
→ observation
```

This is an interface protocol.

It is not yet:

* interpretation
* abstraction
* tactical understanding
* situation recognition
* affordance organization

The environment is therefore better understood as:

> a situation-generation engine.

NetHack continuously generates:

* exploration situations
* escape situations
* uncertainty situations
* combat situations
* resource-collapse situations
* inventory-management situations
* bottleneck situations
* commitment situations

These recurring structures are the actual substrate from which meaningful learning may emerge.

---

# 5. Situation-Centered Learning

The proposed long-term direction differs from simple state-policy mapping.

Traditional RL often emphasizes:

```text id="ekrd0n"
state → action
```

The intended direction here is closer to:

```text id="itv0e1"
situation
→ interpretation
→ affordance organization
→ action
→ outcome
→ sedimented update
```

Human players do not primarily learn exact board states.

They learn recurring structural situations such as:

* being trapped
* overextending
* running low on resources
* having leverage
* losing optionality
* gambling under uncertainty
* controlling a chokepoint
* surviving spike-risk encounters

These are recurring geometries of constrained continuation.

---

# 6. Affordances

IER provides a useful interpretive framework for understanding these structures.

From `IER-affordances.md`:

> “Affordances are locally admissible continuations generated by a system’s constraint organization within an environment.”

Affordances are therefore not:

* static object properties
* globally fixed action menus
* universal gameplay primitives

They are local organizational structures.

A continuation becomes an affordance when it is recruited into the organization of viable futures within a situation.

This means the same environment may generate different affordances depending on:

* danger level
* inventory state
* uncertainty
* narrative context
* resource pressure
* local topology
* current strategic commitments

For example:

```text id="5lq0y4"
door
```

is not intrinsically:

```text id="aax0rx"
openable terrain
```

It may instead become:

* an escape mechanism
* a chokepoint stabilizer
* a visibility barrier
* a risk amplifier
* an irreversible commitment trigger

depending on the current situation.

The affordance field changes as the situation changes.

---

# 7. Situation-Dependent Variables

Variables are also not globally fixed dimensions.

From `IER-variables.md`:

> “A variable exists for a system only when it is recruited into the organization of admissible continuation.”

This is extremely important for NetHack-like environments.

Most gameplay variables only become relevant under specific continuation geometries.

For example:

```text id="uz02fk"
nutrition
```

may barely matter when:

* food is abundant
* retreat paths are open
* exploration pressure is low

but becomes dominant when:

* starving
* trapped
* low HP
* deep in hostile territory

Likewise:

```text id="ry51v7"
scroll identity
```

may not matter at all until uncertainty itself becomes strategically dominant.

This suggests that agents should not treat the environment as a fixed vector space with static feature importance.

Instead:

```text id="tzm06t"
situations recruit variables
```

The situation determines:

* which distinctions matter
* which dimensions organize continuation
* which futures dominate admissibility
* which affordances become salient

Most catastrophic failures in NetHack are arguably failures of variable recruitment.

Examples include:

* not recognizing hunger escalation
* failing to recruit retreat topology
* ignoring uncertainty pressure
* missing irreversible commitment
* failing to recognize affordance collapse

Human players dynamically reorganize which variables exist in practice.

Agents should eventually do the same.

---

# 8. Situation Segmentation

Continuous gameplay dynamics should eventually be segmented into meaningful episodes.

Examples include:

* entering combat
* discovering danger
* losing retreat viability
* exhausting resources
* identifying an item
* committing to exploration
* disengaging from pursuit
* entering a bottleneck

These segmentations matter because they reorganize affordances and variables.

For example:

```text id="vggq4l"
before opening the door
after opening the door
```

are different situations with different reachable futures.

Similarly:

```text id="aijq3v"
before identifying the scroll
after identifying the scroll
```

changes the action-space geometry.

Event segmentation therefore acts as a discretization layer over continuous gameplay.

---

# 9. Situation Clustering

Once segmented, situations may eventually be embedded and clustered.

The goal is not merely compression.

The goal is reusable tactical abstraction.

The system should eventually recognize structures like:

```text id="q4z5ci"
these are all retreat-collapse situations
these are all staircase-search situations
these are all inventory panic situations
these are all controlled exploration expansions
```

This permits transfer learning across superficially different environments.

Different dungeon layouts may still instantiate the same underlying continuation topology.

The important object is not tile similarity.

The important object is situation geometry.

---

# 10. Why NetHack Is Valuable

NetHack naturally produces:

* uncertainty
* stochasticity
* partial observability
* delayed consequences
* dynamic affordance shifts
* inventory-mediated action-space changes
* survival bottlenecks
* long-horizon dependencies
* reversible and irreversible commitments

These properties are useful not merely because they are “hard.”

They are useful because they generate rich recurring situation structures.

NetHack continuously reorganizes local choice space.

This makes it an unusually valuable environment for studying situation-centered learning.

---

# 11. Long-Term Direction

The long-term goal is not merely:

```text id="0ln5e6"
bigger policies
```

or:

```text id="ycv6ys"
benchmark optimization
```

The more interesting direction is toward agents that can:

* segment situations
* interpret situations
* recruit variables dynamically
* recognize affordance structures
* cluster recurring tactical geometries
* generalize across situation families
* reason about constrained continuation

The eventual hope is that this produces agents that behave less like brittle policy optimizers and more like adaptive participants within dynamic worlds.

But that comes later.

The current task remains simple:

```text id="2qk0n5"
reset
step
observe
debug
repeat
```
